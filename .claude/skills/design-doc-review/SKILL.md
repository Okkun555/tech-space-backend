# Design Doc レビュースキル

## 目的

このスキルには2つの目的がある。

1. **Railsの設計学習**：設計判断の質を高め、Railsらしい設計を身につける
2. **書く力の訓練**：「なぜそう設計したか」を構造的に言語化する習慣を作る
   レビューは技術的な正しさだけを見るのではなく、**思考の言語化の質**も必ずフィードバックする。

---

## レビューの観点

### 1. 概要・背景の明確さ
- 何を作るかが一言で伝わるか
- なぜ作るのか（解決する問題）が明確か
- 読んだ人が前提知識なしで理解できるか
### 2. 設計の妥当性（Railsらしさ）
以下の観点でRailsの設計として適切かを確認する。

- **モデル設計**：ActiveRecordの責務が適切か。Fat Modelになっていないか。バリデーション・アソシエーションの設計は妥当か
- **コントローラ設計**：RESTfulなルーティングになっているか。ロジックがコントローラに漏れていないか
- **サービスオブジェクト・その他パターン**：複雑なビジネスロジックの置き場所は適切か
- **データベース設計**：インデックス・外部キー・正規化の判断は妥当か
- **命名**：Railsの慣習に沿った命名になっているか
### 3. 選択肢の検討（最重要）
これが最も重要な観点。Design Docに「検討した他の選択肢」が書かれているかを確認し、以下をフィードバックする。

- 選択肢が書かれていない場合：「他にどんな設計が考えられたか」を提示し、なぜ選んだ案が優れているか（または劣っているか）を問いかける
- 選択肢が書かれている場合：却下理由の妥当性をフィードバックする。「〇〇という理由で却下」に対して、その判断が適切かを評価する
### 4. トレードオフの言語化
- 選んだ設計の弱点・妥協点が書かれているか
- 「完璧な設計」として書いていないか（必ずトレードオフは存在する）
- 将来的な課題として何が残るかを認識しているか
### 5. 未解決の問題
- 実装してみないとわからないことを認識しているか
- 判断を先送りしている部分が明示されているか
---

## フィードバックの形式

以下の構成でフィードバックを返す。

```
## 全体評価
[設計の質と言語化の質をそれぞれ一言で評価]
 
## 良い点
[具体的に何が良かったかを挙げる。抽象的な褒め方はしない]
 
## 設計へのフィードバック
[Railsの設計として改善できる点を具体的に指摘]
 
## 「なぜこの設計か」へのフィードバック
[選択肢の検討とトレードオフの言語化に対するフィードバック。
ここは必ず書く。設計が良くても言語化が不十分なら指摘する]
 
## 検討してほしい別の設計案
[ユーザーが見落としている可能性がある選択肢を1〜2個提示し、
それぞれのメリット・デメリットを示す]
 
## 次のステップ
[実装前に決めておくべきことがあれば指摘]
```
 
---

## レビュー時の姿勢

- **正解を押しつけない**：設計に唯一の正解はない。ユーザーの判断を尊重しながら、思考を深める問いかけをする
- **Railsの慣習を基準にする**：「Railsらしいか」を軸に置く。過度なオーバーエンジニアリングも指摘する
- **言語化の質を必ず評価する**：技術的に正しくても「なぜそうしたか」が書かれていなければ指摘する。これがこのスキルの核心
- **学習段階を考慮する**：Rails歴1年の個人開発という前提で、現時点で習得すべき設計パターンに絞ってフィードバックする。高度すぎる指摘は避ける
---

## テンプレートとの対応

ユーザーは以下のDesign Docテンプレートを使っている。各セクションに対応したフィードバックをする。

```
## 概要
## 背景・目的
## 設計
  ### 選んだアプローチ
  ### 検討した他の選択肢（表形式）
  ### トレードオフ
## 未解決の問題
## 参考
```

「検討した他の選択肢」と「トレードオフ」が空欄または薄い場合は、必ず深掘りする。
